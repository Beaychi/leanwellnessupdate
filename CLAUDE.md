## Architecture Overview

**LeanTrack** is a mobile-first PWA for wellness tracking (meals, exercises, fasting, water, body measurements) built for the Nigerian 7-day diet plan. Pure frontend SPA with an optional Supabase backend for push notifications, AI food analysis, and email reports.

**Stack:** React 18 + TypeScript + Vite (SWC) + Tailwind CSS + shadcn-ui (Radix UI) + React Router v6 + TanStack React Query + Framer Motion + Zustand + Supabase

**Deployment:** Vercel (all routes → `/index.html` via `vercel.json`). Maintained via Lovable.dev which auto-commits to Git.

## Key Structural Patterns

### Two localStorage stores (do not confuse them)

| Key | Type | File | Contents |
|-----|------|------|----------|
| `leantrack_data` | `UserProgress` | `src/lib/storage.ts` | All daily tracking: completed meals, exercise logs, water, weight, streak, settings |
| `leantrack_registration` | `UserRegistration` | `src/lib/registration.ts` | One-time onboarding: name, country, allergies, schedule, email prefs |

`saveStoredData(partial)` merges into the existing record — never pass a full object built from scratch to avoid clobbering unrelated fields. Read before writing anything.

### Dynamic meal plan

`src/lib/meal-plan.ts` stores a `MealPlanData` object under `leantrack_meal_plan`. It is generated by the `generate-meal-plan` edge function based on `UserRegistration` (country + allergies). Components that need today's meals must read this key, not the static `dietPlan.ts`.

### Static diet data vs dynamic meal plan

- `src/lib/dietPlan.ts` — hardcoded 7-day Nigerian meal plan (fallback / demo content)
- `src/lib/meal-plan.ts` — AI-generated plan stored after onboarding; takes precedence
- `src/lib/exercises.ts` — static exercise library

### Timer system

Two timers exist: exercise and fasting. Both share the same Zustand store.

```
src/store/timerStore.ts          — Zustand store with persist middleware
src/lib/active-timer.ts          — Legacy bridge (setActiveTimer / getActiveTimer)
src/hooks/use-timer-display.ts   — RAF-based React hook for smooth UI rendering
src/lib/timer-notifications.ts   — Live system notification (1-second updates, reuses same tag)
```

Time accounting: `elapsedBefore + (Date.now() - startedAt) / 1000`. Never store a countdown; always recompute from timestamps. This keeps the timer accurate after tab switches, background throttling, and page reloads.

Notification action buttons (⏸ Pause | □ Stop | ✕ Cancel) send `TIMER_ACTION` messages from the service worker → `App.tsx` broadcasts a `timerNotificationAction` custom event → `ExerciseTimer` / `FastingTimer` components listen and act.

### Notification system (two separate layers)

1. **Scheduled reminders** (`src/lib/notifications.ts`): Web Notifications API with `setInterval` polling every minute. Plays Web Audio API sounds. Started in `App.tsx` on mount if `notificationsEnabled`.

2. **Push notifications** (`src/lib/push-notifications.ts` + Supabase edge functions): WebPush via service worker. Subscription stored in `push_subscriptions` table. Settings in `notification_settings` table. Edge function `send-scheduled-notifications` handles server-side delivery.

### Service worker

`src/sw.ts` is compiled by `vite-plugin-pwa` (Workbox). It handles:
- Precaching + offline SPA fallback
- Incoming `push` events from the Supabase edge function
- `notificationclick` → focuses the app window

The SW **does not** handle the live timer notification ticks — those come from `timer-notifications.ts` running in the page context and calling `registration.showNotification()`.

### Supabase edge functions

All in `supabase/functions/`. Each is a standalone Deno module. Key functions:

| Function | Purpose |
|----------|---------|
| `analyze-food` | GPT-4 Vision analysis of food photos |
| `generate-meal-plan` | AI-generated 7-day meal plan per user profile |
| `send-push-notification` | Manual/test push send |
| `send-scheduled-notifications` | Cron-triggered delivery of meal/movement reminders |
| `send-email-report` / `send-scheduled-reports` | Weekly/monthly progress emails via Resend |

### Security model (device_id ownership)

No authentication. Ownership uses a `device_id` UUID (stored in `localStorage` as `leantrack_device_id`, attached to every Supabase request as `x-device-id` header in `src/integrations/supabase/client.ts`).

RLS policies read it via `current_setting('request.headers', true)::json ->> 'x-device-id'`.

**Rule:** All inserts to user-data tables must call `getDeviceId()` from `src/lib/push-notifications.ts` — never use `|| 'unknown'` fallbacks. See `SECURITY.md` for full details.

### Routing & app shell

Routes are defined in `src/App.tsx`. `BottomNav` is rendered outside `<Routes>` and is always visible. If `onboardingCompleted` is false or no `UserRegistration` exists, `<Registration>` overlays the entire app.

### Component structure

- `src/components/ui/` — shadcn-ui primitives; regenerate via shadcn CLI, never edit manually
- `src/components/fasting/` — fasting feature (FastingTimer, protocol selector, history)
- `src/components/*.tsx` — feature components (one per domain: CalorieDashboard, ExerciseTimer, WaterTracker, FoodJournal, etc.)
- `src/pages/` — page-level components mapped to routes

### Styling

Tailwind CSS with HSL CSS variables (`--background`, `--primary`, etc.). Dark mode via `class` strategy, toggled with `next-themes`. Custom keyframes in `tailwind.config.ts`. Path alias `@` → `./src`.

### TypeScript config

`noImplicitAny: false`, `strictNullChecks: false` — the project intentionally uses loose settings. IDE TypeScript errors about missing module declarations are pre-existing environment issues; `npm run build` is the authoritative check.
